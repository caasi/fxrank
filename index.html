FxRank — an effect-cost profiler for coding agents

FxRank ranks each function by its effect cost — how much IO, mutation, panic, and risk it carries in its own body — so a coding agent can find the hotspots worth refactoring toward a purer core. It is a measuring instrument: it reports facts and gives no refactoring advice. The decision stays with the agent.

The thesis

FxRank gives a gradient, not a pure/impure label, and it knows a language's type system makes some effects safer than others. A declared, visible effect is bounded and scores lower; the same effect hidden behind a pure-looking signature scores higher. That inversion — the honest cost ranking below the sneaky one — is the whole point.

Imperfect by design, useful anyway

FxRank is primarily syntactic — it runs no type checker and no borrow checker — so it is wrong sometimes. That is acceptable, because it is triage, not a verdict. It only has to be directionally right to float the real hotspots toward the top of a ranking; it need not be precise on every function to save you from reading the whole repo.

The cost is asymmetric: a false positive costs you one look, while a good ranking saves you reading forty files. Treat the output as a signal, not gospel — verify the top hotspots against the source, then decide.

The specific, evolving limitations are not listed here, because they change. They live in the issue tracker: https://github.com/caasi/fxrank/issues.
